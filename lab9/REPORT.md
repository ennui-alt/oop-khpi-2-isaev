\## Опис предметної області



Предметна область лабораторної роботи — система бронювання кімнат.



У такій системі клієнт може вибрати кімнату, створити запис бронювання, а програма може перевірити, чи зайнята кімната на певні дати. Також у програмі можна відокремити підтверджені бронювання від непідтверджених.



\## Опис класів



\### Customer



Клас `Customer` описує клієнта системи бронювання.



Він містить такі поля:



`name` — ім’я клієнта.



`phone` — номер телефону клієнта.



Клас має конструктор, методи доступу `getName()` і `getPhone()`, а також метод `toString()` для зручного виведення інформації про клієнта.



\### Room



Клас `Room` описує кімнату, яку можна забронювати.



Він містить такі поля:



`number` — номер кімнати.



`type` — тип кімнати.



`price` — ціна кімнати.



Клас `Room` реалізує інтерфейс `Comparable<Room>`. Це потрібно для того, щоб кімнати можна було порівнювати між собою за ціною.



Метод `compareTo()` порівнює кімнати за значенням поля `price`.



\### Reservation



Клас `Reservation` описує запис бронювання.



Він містить такі поля:



`customer` — клієнт, який створив бронювання.



`room` — кімната, яку бронюють.



`from` — дата початку бронювання.



`to` — дата завершення бронювання.



`confirmed` — ознака підтвердження бронювання.



Клас має методи:



`getDays()` — рахує кількість днів бронювання.



`getTotalPrice()` — рахує повну вартість бронювання.



`hasConflict()` — перевіряє, чи перетинається бронювання з іншим проміжком дат.



`compareTo()` — порівнює бронювання за датою початку.



Клас `Reservation` реалізує `Comparable<Reservation>`, тому його об’єкти можна використовувати в generic-методі `findSmallest()`.



\## Generic-клас



У роботі реалізовано generic-клас:



`BookingLink<C, R>`



Цей клас має два параметри типу:



`C` — тип клієнта.



`R` — тип ресурсу.



Клас містить два поля:



`customer`



`resource`



Основна ідея класу полягає в тому, що він може типобезпечно зв’язувати клієнта з різними ресурсами.



У демонстраційній програмі використано дві конкретизації цього класу:



`BookingLink<Customer, Room>`



`BookingLink<Customer, Reservation>`



Перша конкретизація показує зв’язок клієнта з кімнатою.



Друга конкретизація показує зв’язок клієнта з готовим записом бронювання.



Таким чином, один generic-клас використовується для різних типів даних, але без втрати типобезпеки.



\## Generic-методи



У класі `BookingUtils` реалізовано кілька generic-методів.



\### Метод getConfirmed



Метод:



`getConfirmed(List<T> reservations)`



має обмеження:



`T extends Reservation`



Цей метод приймає список бронювань і повертає тільки ті записи, які мають `confirmed = true`.



Обмеження `T extends Reservation` потрібне для того, щоб метод міг безпечно викликати метод `isConfirmed()`.



\### Метод isRoomBusy



Метод:



`isRoomBusy(List<T> reservations, Room room, LocalDate from, LocalDate to)`



також має обмеження:



`T extends Reservation`



Метод перевіряє, чи є серед підтверджених бронювань таке бронювання, яке перетинається з переданим проміжком дат для конкретної кімнати.



Цей метод є важливим для предметної області, тому що система бронювання повинна вміти перевіряти зайнятість кімнати.



\### Метод findSmallest



Метод:



`findSmallest(List<T> list)`



має обмеження:



`T extends Comparable<? super T>`



Це означає, що метод може працювати тільки з тими типами, які можна порівнювати між собою.



У роботі цей метод використовується для двох типів:



`Reservation`



`Room`



Для `Reservation` метод знаходить найперше бронювання за датою.



Для `Room` метод знаходить найдешевшу кімнату.



\## Обґрунтування використання generics



У цій роботі використання generics є виправданим, тому що generic-клас `BookingLink<C, R>` справді пов’язаний із предметною областю.



У системі бронювання клієнт може бути пов’язаний не тільки з кімнатою, але й із готовим записом бронювання. Якби для цього використовувався тип `Object`, то довелося б робити приведення типів вручну. Це могло б призвести до помилок під час виконання програми.



Generics дозволяють уникнути таких проблем, тому що типи перевіряються ще під час компіляції.



Також generic-метод `findSmallest()` дозволяє не дублювати код для пошуку мінімального об’єкта. Один і той самий метод може працювати і з кімнатами, і з бронюваннями.

\## Додаткове завдання



У додатковому завданні було розглянуто лабораторну роботу №8 з позиції generics.



У лабораторній роботі №8 була реалізована модель університетських сервісів. Основним базовим класом був `CampusService`, від якого наслідувалися різні типи сервісів: `ExamReminder`, `RoomBooking` і `ConsultationSlot`.



Компонентом, який можна було б узагальнити, є `CampusServiceRegistry`.



У початковій реалізації він зберігає список об’єктів типу:



`List<CampusService>`



Тобто такий реєстр може містити різні типи університетських сервісів одночасно.



Generic-версію цього компонента можна подати так:



```java

public class CampusServiceRegistry<T extends CampusService> {

&#x20;   private final List<T> services = new ArrayList<>();



&#x20;   public void addService(T service) throws InvalidCampusServiceException {

&#x20;       if (service == null) {

&#x20;           throw new InvalidCampusServiceException("Service must not be null");

&#x20;       }



&#x20;       services.add(service);

&#x20;   }



&#x20;   public List<T> getServices() {

&#x20;       return List.copyOf(services);

&#x20;   }



&#x20;   public int size() {

&#x20;       return services.size();

&#x20;   }

}

```



Тоді можна було б створювати окремі типобезпечні реєстри:



```java

CampusServiceRegistry<RoomBooking> roomRegistry = new CampusServiceRegistry<>();

CampusServiceRegistry<ExamReminder> examRegistry = new CampusServiceRegistry<>();

CampusServiceRegistry<CampusService> commonRegistry = new CampusServiceRegistry<>();

```



Наприклад, `CampusServiceRegistry<RoomBooking>` дозволив би зберігати тільки бронювання аудиторій. У такий реєстр не можна було б випадково додати нагадування про іспит або консультацію.



Таке узагальнення покращує API у випадку, коли потрібно працювати з окремими типами сервісів. Але для загального реєстру, де разом зберігаються іспити, консультації та бронювання аудиторій, поточний варіант із `List<CampusService>` також є логічним.



\## Висновок



Під час виконання лабораторної роботи було реалізовано систему бронювання кімнат із використанням generics. Було створено класи `Customer`, `Room` і `Reservation`, які описують основні сутності предметної області. Для зв’язування клієнта з ресурсом реалізовано generic-клас `BookingLink<C, R>`, який використовується у двох конкретизаціях. Також було створено клас `BookingUtils` з generic-методами для відбору, перевірки та вибору записів. Використання generics у цій роботі є доцільним, тому що воно робить код більш універсальним, типобезпечним і не потребує небезпечного приведення типів.
