\# Вимоги до програми



\## Загальні вимоги

1\. Програма повинна бути написана мовою Java.

2\. Програма повинна реалізовувати модель освітнього оцінювання.

3\. У програмі повинні бути використані принципи об'єктно-орієнтованого програмування.

4\. У програмі повинен бути реалізований поліморфізм.

5\. У програмі повинен бути використаний абстрактний клас як узагальнений тип.

6\. Програма повинна відповідати варіанту 3: освітнє оцінювання.



\## Функціональні вимоги

1\. Система повинна зберігати інформацію про оцінювання студента.

2\. Система повинна підтримувати декілька типів оцінювання:

&#x20;  - іспит;

&#x20;  - тестове оцінювання;

&#x20;  - курсовий проєкт.

3\. Система повинна зберігати ідентифікатор оцінювання.

4\. Система повинна зберігати назву навчального курсу.

5\. Система повинна зберігати ім'я студента.

6\. Система повинна зберігати навчальну групу студента.

7\. Система повинна зберігати максимальну кількість балів.

8\. Система повинна зберігати отриману кількість балів.

9\. Система повинна обчислювати відсоток виконання оцінювання.

10\. Система повинна формувати короткий опис оцінювання.

11\. Система повинна визначати результат оцінювання залежно від його типу.

12\. Для іспиту повинна враховуватися наявність прокторингу і мінімальний результат 60%.

13\. Для тестового оцінювання повинна враховуватися кількість спроб і мінімальний результат 50%.

14\. Для курсового проєкту повинна враховуватися наявність захисту і мінімальний результат 70%.



\## Вимоги до поліморфної моделі

1\. Базовий клас `Assessment` повинен бути абстрактним.

2\. У базовому класі `Assessment` повинен бути оголошений абстрактний метод `evaluateResult()`.

3\. Базовий клас повинен містити спільний стан для всіх типів оцінювання.

4\. Базовий клас повинен містити спільну поведінку для обчислення відсотка і формування опису.

5\. Класи `Exam`, `QuizAssessment` і `CourseProject` повинні наслідувати клас `Assessment`.

6\. Кожен клас-нащадок повинен перевизначати метод `evaluateResult()`.

7\. Для перевизначених методів повинна використовуватися анотація `@Override`.

8\. Поліморфний виклик методу повинен виконуватись через посилання типу `Assessment`.

9\. У програмі має бути продемонстровано, що різні об'єкти по-різному реагують на однаковий виклик методу `evaluateResult()`.



\## Технічні вимоги

1\. Для демонстрації поліморфізму використовується масив `Assessment\[]`.

2\. Проєкт повинен містити клас `Main`, який демонструє роботу програми.

3\. Для тестування використовуються JUnit 5.

4\. Проєкт повинен містити юніт-тести для спільного контракту базового класу.

5\. Проєкт повинен містити юніт-тести для специфічної поведінки підкласів.

6\. Проєкт повинен містити UML-діаграму класів у форматі PlantUML.

7\. Проєкт повинен містити файл README з описом роботи.

8\. Проєкт повинен містити файл REPORT з коротким поясненням реалізації.

9\. Проєкт повинен збиратися за допомогою Maven.

10\. Проєкт повинен успішно проходити команду `mvn test`.
