\# Звіт з лабораторної роботи №11



\## Тема



Collections Framework в Java.



\## Мета роботи



Метою лабораторної роботи є формування практичних навичок використання стандартних колекцій Java Collections Framework.



Під час виконання роботи потрібно навчитися обирати відповідний тип колекції для конкретного завдання, зберігати об’єкти предметної області, виконувати основні операції над ними та пояснювати, чому було використано саме такі структури даних.



\## Постановка завдання



Потрібно розробити консольний Java-застосунок для заданої предметної області.



У роботі необхідно:



\- створити модель предметної області щонайменше з двох або трьох класів;

\- реалізувати клас-сервіс або клас-реєстр, який використовує стандартні колекції Java;

\- використати щонайменше дві різні колекції з-поміж `List`, `Set`, `Map`, `Queue`;

\- реалізувати базові операції додавання, перегляду, пошуку, оновлення або видалення елементів;

\- продемонструвати обхід колекцій;

\- підготувати демонстраційну програму в `Main`;

\- написати модульні тести для ключових операцій.



\## Вибір варіанта



Номер у списку групи:



```text

N = 3

```



Номер варіанта визначається за формулою:



```text

V = ((N - 1) mod 15) + 1

```



Підставляємо значення:



```text

V = ((3 - 1) mod 15) + 1 = 3

```



Отже, індивідуальний варіант — \*\*№3\*\*.



Тема варіанта — \*\*планувальник завдань команди\*\*.

\## Опис предметної області



У лабораторній роботі реалізовано сервіс планування задач команди розробників.



Такий сервіс може використовуватися для збереження задач, перегляду списку задач, пошуку задачі за ідентифікатором, зміни статусу задачі, видалення задачі та роботи з чергою задач на виконання.



У програмі кожна задача має:



\- ідентифікатор;

\- назву;

\- опис;

\- пріоритет;

\- відповідального розробника;

\- спринт;

\- статус.



Розробник має ім’я, електронну пошту та спеціалізацію. Спринт має назву, дату початку та дату завершення.



\## Архітектура програми



Програма поділена на кілька логічних частин:



\- `model` — класи предметної області;

\- `service` — сервісний клас для роботи з колекціями;

\- `demo` — демонстраційна програма;

\- `test` — модульні тести;

\- `docs` — UML-діаграма.



Основна логіка роботи з колекціями зосереджена в класі `TeamTaskPlanner`. Це дозволяє відокремити класи предметної області від логіки зберігання, пошуку, оновлення та видалення даних.

\## Класи предметної області



\### Клас `Developer`



Клас `Developer` описує розробника команди.



Основні поля класу:



```java

private String name;

private String email;

private String specialization;

```



Поле `email` використовується як унікальна ознака розробника. Саме тому методи `equals()` та `hashCode()` реалізовані на основі електронної пошти.



Це потрібно для коректної роботи колекції `Set<Developer>`, яка не повинна зберігати однакових розробників.



\### Клас `Sprint`



Клас `Sprint` описує спринт, у межах якого виконуються задачі.



Основні поля класу:



```java

private String name;

private String startDate;

private String endDate;

```



Об’єкт `Sprint` використовується в класі `Task`, щоб показати, до якого спринту належить конкретна задача.



\### Клас `Task`



Клас `Task` описує задачу команди.



Основні поля класу:



```java

private String id;

private String title;

private String description;

private int priority;

private Developer developer;

private Sprint sprint;

private TaskStatus status;

```



Поле `id` використовується як унікальний ідентифікатор задачі. На його основі реалізовано методи `equals()` та `hashCode()`.



Кожна задача також має відповідального розробника, спринт і статус.



\### Перелік `TaskStatus`



Перелік `TaskStatus` описує можливі статуси задачі:



```java

TODO

IN\_PROGRESS

DONE

```



Статус задачі може змінюватися під час роботи програми.



\## Сервісний клас `TeamTaskPlanner`



Клас `TeamTaskPlanner` є центральним класом програми. Він виконує роль сервісу, який керує колекціями та надає методи для роботи з даними.



У класі використано такі колекції:



```java

private final List<Task> tasks = new ArrayList<>();

private final Queue<Task> taskQueue = new LinkedList<>();

private final Set<Developer> developers = new LinkedHashSet<>();

```



\## Обґрунтування вибору колекцій



\### `List<Task>`



Колекція `List<Task>` використовується як основне внутрішнє сховище задач.



Вона підходить для цієї роботи, тому що:



\- зберігає задачі у порядку додавання;

\- дозволяє переглядати всі задачі;

\- дозволяє виконувати послідовний пошук;

\- зручна для сортування задач за пріоритетом.



У програмі `List<Task>` використовується в методах:



```java

addTask()

findTaskById()

findTasksByDeveloper()

removeTask()

getAllTasks()

getTasksSortedByPriority()

```



\### `Queue<Task>`



Колекція `Queue<Task>` використовується для організації черги задач на виконання.



Задачі обробляються за принципом FIFO:



```text

First In — First Out

```



Це означає, що задача, яка була додана першою, першою потрапляє на виконання.



У програмі для додавання задачі до черги використовується метод:



```java

offer()

```



Для отримання наступної задачі використовується метод:



```java

poll()

```



\### `Set<Developer>`



Колекція `Set<Developer>` використовується для збереження унікальних розробників команди.



Ця колекція підходить для задачі, тому що один і той самий розробник не повинен додаватися повторно.



Для коректної роботи `Set` у класі `Developer` реалізовано методи:



```java

equals()

hashCode()

```



Порівняння розробників виконується за полем `email`.



\## Основні операції сервісу



У класі `TeamTaskPlanner` реалізовано основні операції для роботи з даними.



\### Додавання



Для додавання даних використовуються методи:



```java

addTask(Task task)

addDeveloper(Developer developer)

```



Метод `addTask()` додає задачу до основного списку `List<Task>`, до черги `Queue<Task>`, а також додає розробника задачі до множини `Set<Developer>`.



\### Пошук



Для пошуку використовуються методи:



```java

findTaskById(String id)

findTasksByDeveloper(Developer developer)

```



Метод `findTaskById()` шукає задачу за її унікальним ідентифікатором.



Метод `findTasksByDeveloper()` повертає список задач, які належать конкретному розробнику.



\### Перегляд



Для перегляду даних використовуються методи:



```java

getAllTasks()

getDevelopers()

printAllTasks()

printDevelopers()

```



Для перегляду задач використовується обхід через `for-each`.



Для перегляду розробників використовується `Iterator`.



\### Оновлення



Для оновлення статусу задачі використовується метод:



```java

updateTaskStatus(String id, TaskStatus status)

```



Метод знаходить задачу за `id` і змінює її статус.



\### Видалення



Для видалення задачі використовується метод:



```java

removeTask(String id)

```



Метод видаляє задачу з основного списку `List<Task>` та з черги `Queue<Task>`.



\## Підтримка узгодженості внутрішнього стану



У сервісі одна операція може оновлювати кілька колекцій.



Під час додавання задачі виконуються такі дії:



```java

tasks.add(task);

taskQueue.offer(task);

developers.add(task.getDeveloper());

```



Це означає, що задача потрапляє до списку всіх задач, до черги виконання, а її розробник додається до множини розробників.



Під час видалення задачі виконуються такі дії:



```java

tasks.remove(task);

taskQueue.remove(task);

```



Задача видаляється і з основного списку, і з черги. Завдяки цьому внутрішній стан сервісу залишається узгодженим.



\## Додаткове завдання



У роботі виконано частину додаткового завдання.



\### Сортування через `Comparator`



Для сортування задач за пріоритетом реалізовано метод `getTasksSortedByPriority()`.



```java

public List<Task> getTasksSortedByPriority() {

&#x20;   List<Task> sortedTasks = new ArrayList<>(tasks);

&#x20;   sortedTasks.sort(Comparator.comparingInt(Task::getPriority));

&#x20;   return sortedTasks;

}

```



Метод створює копію списку задач і сортує її за пріоритетом. Основний список задач при цьому не змінюється.



\### Усунення дублікатів через `Set`



Для збереження розробників використано `Set<Developer>`.



Завдяки цьому один і той самий розробник не додається повторно.



\### Черга задач через `Queue`



Для організації черги задач використано `Queue<Task>`.



Метод `takeNextTask()` повертає наступну задачу з черги:



```java

public Task takeNextTask() {

&#x20;   return taskQueue.poll();

}

```



\### Порівняння можливих реалізацій



Для збереження розробників можна було використати `List<Developer>`, але тоді потрібно було б вручну перевіряти дублікати.



Тому було обрано `Set<Developer>`, оскільки ця колекція автоматично не допускає повторення однакових елементів.



Для зберігання задач можна було використати тільки `Queue<Task>`, але черга не є зручною для повного перегляду, пошуку та сортування.



Тому основним сховищем задач обрано `List<Task>`, а `Queue<Task>` використано окремо саме для моделювання черги виконання задач.
