\## Короткий опис побудованої предметної моделі



У лабораторній роботі було побудовано модель інформаційних сервісів кампусу. Ця модель описує сервіси, які можуть бути корисними студенту під час навчання: нагадування про іспит, бронювання аудиторії та консультація з викладачем.



Основними класами предметної області є `ExamReminder`, `RoomBooking` і `ConsultationSlot`.



Клас `ExamReminder` описує нагадування про іспит. Він містить інформацію про студента, дату, час, назву дисципліни та аудиторію.



Клас `RoomBooking` описує бронювання аудиторії. Він містить інформацію про студента, дату, час, номер аудиторії та мету бронювання.



Клас `ConsultationSlot` описує консультацію з викладачем. Він містить інформацію про студента, дату, час, ім’я викладача та тему консультації.



Також було створено абстрактний клас `CampusService`. У ньому винесено спільні поля для всіх сервісів: `id`, `studentName`, `date` і `time`.



Це дозволяє не дублювати однакові поля у кожному класі. Класи `ExamReminder`, `RoomBooking` і `ConsultationSlot` наслідуються від `CampusService` і додають власні поля, які відповідають їхньому призначенню.



\## Пояснення ролі класу-контейнера



Роль класу-контейнера виконує клас `CampusServiceRegistry`.



Цей клас зберігає список сервісів кампусу в пам’яті програми. Для цього використовується колекція `List<CampusService>`. Оскільки всі конкретні сервіси наслідуються від `CampusService`, у цьому списку можна зберігати різні типи об’єктів: `ExamReminder`, `RoomBooking` і `ConsultationSlot`.



У класі `CampusServiceRegistry` реалізовано такі основні методи:



\- `addService()` — додає новий сервіс до реєстру;

\- `getServices()` — повертає список усіх сервісів;

\- `findById()` — виконує пошук сервісу за ідентифікатором;

\- `findByStudentName()` — виконує пошук сервісів за іменем студента;

\- `findByDate()` — виконує пошук сервісів за датою;

\- `size()` — повертає кількість записів у реєстрі.



Також контейнер перевіряє дублювання ідентифікаторів і конфлікт записів. Наприклад, якщо студент уже має подію на певну дату і час, програма не дозволяє додати іншу подію на той самий час.



Таким чином, `CampusServiceRegistry` відповідає за збереження об’єктів у пам’яті, додавання, перегляд і пошук записів.



\## Опис текстового формату збереження даних



Для збереження даних у текстовий файл було визначено простий формат. Кожен об’єкт записується в один рядок. Поля розділяються символом `;`.



Загальний формат рядка:



```text

TYPE;id;studentName;date;time;field1;field2

```



Поле `TYPE` визначає тип запису. У програмі використовуються такі типи:



```text

EXAM

ROOM

CONSULTATION

```



Для класу `ExamReminder` використовується такий формат:



```text

EXAM;id;studentName;date;time;courseName;auditorium

```



Приклад:



```text

EXAM;E001;Zakhar Isaiev;2026-05-20;10:30;Object-Oriented Programming;301

```



Для класу `RoomBooking` використовується такий формат:



```text

ROOM;id;studentName;date;time;roomNumber;purpose

```



Приклад:



```text

ROOM;R001;Zakhar Isaiev;2026-05-21;14:00;204;Team project meeting

```



Для класу `ConsultationSlot` використовується такий формат:



```text

CONSULTATION;id;studentName;date;time;teacherName;subject

```



Приклад:



```text

CONSULTATION;C001;Zakhar Isaiev;2026-05-22;12:15;Valentina V.;Java serialization

```



Обов’язковими полями для всіх записів є тип сервісу, ідентифікатор, ім’я студента, дата і час. Також обов’язковими є два додаткові поля, які залежать від конкретного типу запису.



Під час читання текстового файлу програма перевіряє кількість полів у рядку. У правильному рядку має бути 7 полів. Якщо кількість полів неправильна або тип запису невідомий, виникає виняткова ситуація.



\## Файли, які створюються під час роботи програми



Після запуску класу `Main` програма створює такі файли:



```text

build/lab08/campus-services.txt

build/lab08/campus-services.bin

build/lab08/broken-campus-services.txt

```



Файл `campus-services.txt` використовується для збереження даних у текстовому форматі.



Файл `campus-services.bin` використовується для серіалізації реєстру у бінарний файл.



Файл `broken-campus-services.txt` використовується для демонстрації помилкової ситуації під час читання неправильного формату даних.



\## Пояснення, як виконано серіалізацію і десеріалізацію



Серіалізація і десеріалізація реалізовані в окремому класі `CampusServiceFileManager`.



Метод `serialize()` відповідає за запис об’єкта `CampusServiceRegistry` у бінарний файл. Для цього використовується `ObjectOutputStream`. Перед записом створюється потрібна директорія, якщо її ще немає.



Метод `deserialize()` відповідає за читання об’єкта `CampusServiceRegistry` з бінарного файлу. Для цього використовується `ObjectInputStream`.



Щоб об’єкти можна було серіалізувати, класи моделі реалізують інтерфейс `Serializable`. Це дозволяє Java перетворювати об’єкти на послідовність байтів і зберігати їх у файлі.



У роботі серіалізація реалізована окремо від текстового формату. Текстовий файл використовується для людиночитного збереження даних, а бінарний файл — для швидкого збереження і відновлення всього стану реєстру.



\## Опис винятків, які можуть виникати під час роботи програми



Під час роботи програми можуть виникати декілька типів винятків.



`IOException` може виникнути під час роботи з файлами. Наприклад, якщо файл не вдалося створити, записати або прочитати. У програмі цей виняток може виникати в методах `saveAsText()`, `loadFromText()`, `serialize()` і `deserialize()`.



`ClassNotFoundException` може виникнути під час десеріалізації бінарного файлу. Це можливо тоді, коли у файлі збережено об’єкт класу, якого немає у поточній програмі, або якщо структура класів була змінена.



`InvalidCampusServiceException` — це користувацький виняток предметної області. Він використовується для помилок, пов’язаних із сервісами кампусу. Наприклад, якщо у текстовому файлі неправильна кількість полів, якщо тип запису невідомий, якщо повторюється `id` запису або якщо студент має два сервіси на одну дату й один час.



`IllegalArgumentException` може виникнути під час створення об’єктів моделі, якщо передані неправильні значення. Наприклад, порожній `id`, порожнє ім’я студента, відсутня дата, відсутній час, порожня назва дисципліни, аудиторії, викладача або теми консультації.



Також може виникнути помилка при перетворенні дати або часу, якщо у текстовому файлі дата чи час записані в неправильному форматі.



У програмі винятки обробляються через блоки `try-catch`. Завдяки цьому програма не завершується аварійно, а виводить зрозуміле повідомлення про причину помилки.



\## Приклад помилкової ситуації та спосіб її обробки



У програмі в класі `Main` показано приклад помилкової ситуації. Для цього створюється текстовий файл з неправильним рядком:



```text

EXAM;BAD\_LINE;Zakhar Isaiev

```



Цей рядок є неправильним, тому що в ньому тільки 3 поля замість 7. Правильний запис повинен містити тип запису, ідентифікатор, ім’я студента, дату, час і два додаткові поля.



Під час читання такого файлу метод `loadFromText()` визначає неправильний формат рядка і генерує `InvalidCampusServiceException`.



У класі `Main` цей виняток перехоплюється в блоці `try-catch`. Програма не завершується аварійно, а виводить повідомлення:



```text

Error example was handled:

Invalid data in line 1: EXAM;BAD\_LINE;Zakhar Isaiev

```



Це показує, що помилкова ситуація обробляється коректно.

\## Додаткове завдання



Для додаткового завдання було обрано модель з попередньої лабораторної роботи №7. У тій роботі розглядалася предметна область інформаційних сервісів кампусу. Основними класами моделі були `ExamReminder`, `RoomBooking` і `ConsultationSlot`. Також використовувалися інтерфейси `Notifiable` і `Schedulable`, які описували можливість повідомлення студента та планування події.



Цю модель можна адаптувати до зовнішнього збереження даних, тому що її об’єкти мають прості поля, які можна записати у текстовий файл. Для цього доцільно використати формат, у якому кожен об’єкт зберігається в окремому рядку, а поля розділяються символом `;`.



Запропонований текстовий формат збереження:



```text

EXAM;id;studentName;date;time;courseName;auditorium

ROOM;id;studentName;date;time;roomNumber;purpose

CONSULTATION;id;studentName;date;time;teacherName;subject

```



Наприклад:



```text

EXAM;E001;Zakhar Isaiev;2026-05-20;10:30;Object-Oriented Programming;301

ROOM;R001;Zakhar Isaiev;2026-05-21;14:00;204;Team project meeting

CONSULTATION;C001;Zakhar Isaiev;2026-05-22;12:15;Savchenko V.;Java serialization

```



Обов’язковими полями для всіх записів є тип сервісу, ідентифікатор, ім’я студента, дата та час. Для `ExamReminder` додатково обов’язковими є назва дисципліни та аудиторія. Для `RoomBooking` обов’язковими є номер аудиторії та мета бронювання. Для `ConsultationSlot` обов’язковими є ім’я викладача та тема консультації.



Можливими помилками формату є неправильна кількість полів у рядку, порожній ідентифікатор, відсутнє ім’я студента, неправильний формат дати або часу, невідомий тип запису, а також конфліктні записи, коли один студент має дві події на одну дату й один час.



Серіалізація всієї колекції об’єктів для цієї моделі є доцільною. Вона дозволяє швидко зберегти повний стан реєстру сервісів кампусу у бінарний файл і потім відновити його без ручного розбору кожного поля. Але серіалізація не повинна повністю замінювати текстовий формат, тому що текстовий файл зручніший для перегляду, перевірки та ручного редагування.

\## Висновок



У ході лабораторної роботи №8 було розроблено Java-застосунок для роботи з інформаційними сервісами кампусу. Було створено модель предметної області з класами `ExamReminder`, `RoomBooking` і `ConsultationSlot`, а також базовий клас `CampusService` для спільних полів. Для зберігання об’єктів у пам’яті використано клас-контейнер `CampusServiceRegistry`, а файлові операції винесено в окремий клас `CampusServiceFileManager`.



У програмі реалізовано запис даних у текстовий файл, читання з текстового файлу, серіалізацію та десеріалізацію реєстру у бінарний файл. Також було створено власний виняток `InvalidCampusServiceException` і показано обробку помилкової ситуації під час читання некоректного рядка з файлу.



У результаті роботи було закріплено навички файлового введення-виведення, серіалізації об’єктів та обробки винятків у Java.
