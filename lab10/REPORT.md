\# Звіт до лабораторної роботи №10



\## Тема роботи



Узагальнені контейнери та ітератори в Java.



\## Варіант



Варіант 3 — \*\*Список задач команди\*\*.



\## Мета роботи



Метою лабораторної роботи є формування практичних навичок створення власного узагальненого контейнера, реалізації власного ітератора та організації послідовного обходу елементів контейнера в Java.



У роботі потрібно було створити класи предметної області, реалізувати generic-контейнер, забезпечити підтримку `Iterable<T>`, продемонструвати роботу контейнера в класі `Main` та написати юніт-тести для перевірки контейнера й ітератора.



\## Постановка завдання



Відповідно до варіанта 3 потрібно реалізувати предметну область \*\*“Список задач команди”\*\*.



Необхідно створити класи:



\- `Task` — задача команди;

\- `Developer` — розробник;

\- `StatusRecord` — запис про зміну статусу задачі.



Також потрібно реалізувати власний узагальнений контейнер для зберігання задач і механізм їх послідовного обходу.



Основні вимоги до роботи:



\- створити власний generic-контейнер;

\- реалізувати власний ітератор;

\- забезпечити підтримку `for-each` через `Iterable<T>`;

\- не використовувати `ArrayList` як основне внутрішнє сховище;

\- продемонструвати додавання, доступ, видалення та обхід елементів;

\- написати JUnit-тести для контейнера та ітератора;

\- підготувати UML-діаграму класів.



\## Опис предметної області



Предметна область лабораторної роботи — список задач команди.



У такій системі команда розробників працює над набором задач. Кожна задача має назву, опис, відповідального розробника та поточний статус. У процесі роботи статус задачі може змінюватися, тому для задачі також зберігається історія зміни статусів.



Наприклад, задача може спочатку мати статус `NEW`, потім перейти в `IN\_PROGRESS`, після цього в `CHECKING`, а в кінці — в `DONE`.



Для опису цієї предметної області були створені такі класи:



\- `Developer`;

\- `Task`;

\- `StatusRecord`;

\- `TaskStatus`.



\## Опис класу Developer



Клас `Developer` описує розробника команди.



Він містить такі поля:



```java

private int id;

private String name;

private String role;

```



Поле `id` зберігає ідентифікатор розробника.



Поле `name` зберігає ім’я розробника.



Поле `role` зберігає роль розробника в команді, наприклад `Backend developer`, `Frontend developer` або `Tester`.



Клас має конструктор:



```java

public Developer(int id, String name, String role)

```



Також у класі реалізовано getter-методи:



```java

getId()

getName()

getRole()

```



Метод `toString()` використовується для зручного виведення інформації про розробника в консоль.



\## Опис enum TaskStatus



Перелік `TaskStatus` описує можливі статуси задачі.



У роботі використано такі статуси:



```java

NEW,

IN\_PROGRESS,

CHECKING,

DONE

```



Статус `NEW` означає, що задача створена.



Статус `IN\_PROGRESS` означає, що задача знаходиться в роботі.



Статус `CHECKING` означає, що задача передана на перевірку.



Статус `DONE` означає, що задача виконана.



Використання `enum` є зручним, тому що статус задачі може мати тільки один із наперед визначених варіантів.



\## Опис класу StatusRecord



Клас `StatusRecord` описує один запис про зміну статусу задачі.



Він містить такі поля:



```java

private TaskStatus status;

private String comment;

private LocalDateTime date;

private Developer developer;

```



Поле `status` зберігає статус, який був встановлений для задачі.



Поле `comment` містить короткий коментар до зміни статусу.



Поле `date` зберігає дату й час створення запису.



Поле `developer` показує, який розробник змінив статус задачі.



Конструктор класу має такий вигляд:



```java

public StatusRecord(TaskStatus status, String comment, Developer developer)

```



Дата створення запису встановлюється автоматично за допомогою:



```java

LocalDateTime.now()

```



Це дозволяє фіксувати момент зміни статусу задачі.



\## Опис класу Task



Клас `Task` описує задачу команди.



Він містить такі основні поля:



```java

private int id;

private String title;

private String description;

private Developer developer;

private TaskStatus status;

```



Поле `id` зберігає номер задачі.



Поле `title` зберігає назву задачі.



Поле `description` зберігає короткий опис задачі.



Поле `developer` містить розробника, відповідального за задачу.



Поле `status` зберігає поточний статус задачі.



Також у класі є поля для зберігання історії статусів:



```java

private StatusRecord\[] history;

private int historySize;

```



Масив `history` зберігає записи про зміну статусів задачі.



Поле `historySize` зберігає кількість фактично доданих записів історії.



При створенні задачі автоматично встановлюється статус `NEW` і додається перший запис історії:



```java

addStatusRecord(TaskStatus.NEW, "Task was created", developer);

```



Для зміни статусу задачі використовується метод:



```java

public void changeStatus(TaskStatus newStatus, String comment, Developer developer)

```



Цей метод змінює поточний статус задачі та додає новий запис до історії статусів.



\## Реалізований контейнер



Для зберігання задач реалізовано власний узагальнений контейнер:



```java

public class TaskContainer<T> implements Iterable<T>

```



Контейнер є generic-класом, тому може працювати з різними типами даних.



У цій лабораторній роботі він використовується для зберігання об’єктів типу `Task`.



Приклад використання:



```java

TaskContainer<Task> taskList = new TaskContainer<>(2);

```



Generic-підхід дозволяє зробити контейнер універсальним і типобезпечним. Це означає, що контейнер може зберігати об’єкти конкретного типу без потреби в ручному приведенні типів.



\## Внутрішнє подання контейнера



Внутрішнє сховище контейнера реалізовано за допомогою звичайного масиву:



```java

private T\[] data;

```



Кількість фактично доданих елементів зберігається у полі:



```java

private int size;

```



Масив `data` відповідає за фізичне зберігання елементів.



Поле `size` відповідає за логічну кількість елементів у контейнері.



Стандартні колекції Java, наприклад `ArrayList`, у контейнері не використовуються як основне сховище. Це відповідає вимозі лабораторної роботи, тому що контейнер повинен бути самописним.



\## Основні методи контейнера



\### Метод add()



Метод `add()` додає новий елемент у контейнер.



```java

public void add(T item)

```



Якщо внутрішній масив уже заповнений, перед додаванням нового елемента викликається метод `grow()`.



Це дозволяє контейнеру автоматично збільшувати свою місткість.



\### Метод get()



Метод `get()` повертає елемент за індексом.



```java

public T get(int index)

```



Перед поверненням елемента виконується перевірка індексу за допомогою методу `checkIndex()`.



Якщо індекс неправильний, генерується виняток:



```java

IndexOutOfBoundsException

```



\### Метод remove()



Метод `remove()` видаляє елемент за індексом.



```java

public T remove(int index)

```



Після видалення елемента всі елементи, які стояли праворуч від нього, зсуваються на одну позицію вліво.



Це потрібно для того, щоб у масиві не залишалося порожнього місця між елементами.



Метод також повертає видалений елемент.



\### Метод size()



Метод `size()` повертає кількість елементів у контейнері.



```java

public int size()

```



\### Метод capacity()



Метод `capacity()` повертає поточну місткість внутрішнього масиву.



```java

public int capacity()

```



Цей метод зручно використовувати для демонстрації автоматичного розширення масиву.



\### Метод isEmpty()



Метод `isEmpty()` перевіряє, чи є контейнер порожнім.



```java

public boolean isEmpty()

```



Якщо `size` дорівнює нулю, метод повертає `true`.



\## Додаткове завдання



Як додаткове завдання було реалізовано автоматичне розширення внутрішнього масиву.



За це відповідає метод:



```java

private void grow()

```



Якщо під час додавання нового елемента кількість елементів дорівнює довжині масиву, створюється новий масив у два рази більшого розміру:



```java

T\[] newData = (T\[]) new Object\[data.length \* 2];

```



Після цього всі елементи зі старого масиву копіюються в новий масив:



```java

for (int i = 0; i < data.length; i++) {

&#x20;   newData\[i] = data\[i];

}

```



Потім посилання `data` змінюється на новий масив:



```java

data = newData;

```



У демонстраційній програмі створюється контейнер з початковою місткістю `2`, після чого до нього додаються три задачі. У результаті місткість контейнера автоматично збільшується до `4`.



Приклад результату:



```text

Size: 3

Capacity: 4

```



Це підтверджує, що додаткове завдання виконано.



\## Реалізація ітератора



Для послідовного обходу контейнера реалізовано внутрішній клас:



```java

private class TaskIterator implements Iterator<T>

```



Ітератор містить поле:



```java

private int current = 0;

```



Поле `current` зберігає індекс поточного елемента під час обходу.



Клас `TaskIterator` реалізує два основні методи:



```java

public boolean hasNext()

public T next()

```



Метод `hasNext()` перевіряє, чи існує наступний елемент:



```java

return current < size;

```



Метод `next()` повертає наступний елемент контейнера і збільшує індекс `current`.



Якщо наступного елемента немає, метод `next()` генерує виняток:



```java

NoSuchElementException

```



Таким чином, ітератор дозволяє послідовно переглядати всі елементи контейнера без прямого доступу до внутрішнього масиву.



\## Підтримка for-each



Контейнер реалізує інтерфейс:



```java

Iterable<T>

```



Для цього в класі `TaskContainer` перевизначено метод:



```java

@Override

public Iterator<T> iterator()

```



Метод `iterator()` повертає новий об’єкт внутрішнього ітератора:



```java

return new TaskIterator();

```



Завдяки цьому контейнер можна обходити за допомогою циклу `for-each`.



Приклад:



```java

for (Task task : taskList) {

&#x20;   System.out.println(task);

}

```



Це демонструє, що контейнер підтримує стандартний механізм послідовного обходу елементів у Java.



\## Демонстраційна програма Main



У класі `Main` продемонстровано роботу всіх основних можливостей програми.



Спочатку створюються три розробники:



```java

Developer dev1 = new Developer(1, "Zakhar Isaiev", "Backend developer");

Developer dev2 = new Developer(2, "Oleg Vinnik", "Frontend developer");

Developer dev3 = new Developer(3, "Valentin Strikalo", "Tester");

```



Потім створюються три задачі:



```java

Task task1 = new Task(1, "Authorization", "Create login and registration", dev1);

Task task2 = new Task(2, "Main page", "Create main page layout", dev2);

Task task3 = new Task(3, "Testing", "Check task module", dev3);

```



Після цього створюється контейнер:



```java

TaskContainer<Task> taskList = new TaskContainer<>(2);

```



До контейнера додаються три задачі:



```java

taskList.add(task1);

taskList.add(task2);

taskList.add(task3);

```



Це демонструє додавання елементів і автоматичне розширення масиву.



У програмі також показано:



\- доступ до задач за індексом;

\- зміну статусів задач;

\- обхід через `Iterator`;

\- обхід через `for-each`;

\- перегляд історії статусів;

\- видалення задачі з контейнера.

Після запуску програми виводиться інформація про додавання задач до контейнера.



Приклад результату:



```text

Adding tasks to container...

```



Цей результат показує, що в контейнер додано три задачі, а місткість автоматично збільшилася з `2` до `4`.



Далі програма виводить задачі за індексом, демонструє зміну статусів, обхід через ітератор і обхід через `for-each`.

\## Чому обраний контейнер і його API є коректними для задачі



Для задачі списку командних задач контейнер `TaskContainer<T>` є коректним, тому що він дозволяє зберігати набір задач і виконувати над ними базові операції.



Для списку задач важливо мати можливість:



\- додати нову задачу;

\- отримати задачу за позицією;

\- видалити задачу;

\- дізнатися кількість задач;

\- перевірити, чи список порожній;

\- послідовно переглянути всі задачі.



Усі ці операції реалізовані в API контейнера.



Контейнер є generic, тому його можна використовувати не тільки для `Task`, але й для інших типів. Це робить реалізацію більш універсальною і типобезпечною.



Використання власного ітератора дозволяє відокремити внутрішнє зберігання елементів від способу їх обходу. Клієнтський код не повинен знати, що всередині контейнера використовується масив.

У ході виконання лабораторної роботи було створено модель предметної області “Список задач команди”.

Під час виконання роботи було краще засвоєно принципи роботи generic-контейнерів, ітераторів, інтерфейсів `Iterator<T>` та `Iterable<T>` у Java.
