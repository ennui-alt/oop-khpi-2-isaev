\## Ідея роботи



У цій лабораторній роботі модель побудована навколо інтерфейсів, які описують поведінку об’єктів.



У програмі є сервіси кампусу, які можуть повідомляти студента або планувати подію на певний час.



У проєкті введено два інтерфейси:



\- `Notifiable` — для об’єктів, які можуть повідомляти студента;

\- `Schedulable` — для об’єктів, які можна запланувати.



\## Які класи реалізують інтерфейси



\- `ExamReminder` реалізує `Notifiable`;

\- `RoomBooking` реалізує `Schedulable`;

\- `ConsultationSlot` реалізує `Notifiable` і `Schedulable`.



Таким чином:



\- у моделі є два інтерфейси;

\- у моделі є три конкретні класи;

\- клас `ConsultationSlot` реалізує більше ніж один інтерфейс.



\## Що показано в роботі



У роботі показано, що різні класи можуть працювати через один спільний інтерфейс.



У `Main` використовується масив `Notifiable\[]`, через який обробляються об’єкти, що можуть повідомляти студента.



Також використовується масив `Schedulable\[]`, через який обробляються об’єкти, які можна запланувати.



Один і той самий об’єкт `ConsultationSlot` використовується через два різні інтерфейсні подання:



\- як `Notifiable`;

\- як `Schedulable`.
