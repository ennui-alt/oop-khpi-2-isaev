\# REPORT.md



\# Лабораторна робота №15



\## Тема



Синхронізація потоків у Java.



\## Мета роботи



Мета лабораторної роботи — отримати практичні навички роботи з багатопотоковістю в Java, навчитися виявляти проблему спільного доступу до змінюваних даних, визначати критичні секції та використовувати механізм `synchronized` для коректної роботи потоків зі спільним ресурсом.



\## Варіант



\*\*Варіант 3 — бронювання номерів.\*\*



За індивідуальним варіантом необхідно реалізувати предметну область, пов’язану з бронюванням номерів. У програмі кілька потоків одночасно намагаються забронювати обмежену кількість місць у готелі.



\## Короткі теоретичні відомості



У багатопотокових програмах кілька потоків можуть одночасно звертатися до одного й того самого об’єкта. Якщо цей об’єкт містить змінюваний стан, виникає спільний змінюваний ресурс.



Проблема виникає тоді, коли кілька потоків одночасно читають, змінюють і записують одне й те саме значення. Така ситуація називається станом перегонів. Результат програми в такому випадку залежить від порядку виконання потоків, який не контролюється програмістом.



Критична секція — це частина програми, яку не повинні одночасно виконувати кілька потоків, якщо вона змінює спільний ресурс.



У Java для захисту критичних секцій можна використовувати ключове слово `synchronized`. Воно дозволяє зробити так, щоб у певний момент часу тільки один потік міг виконувати синхронізований метод або блок коду.



\## Постановка індивідуального завдання



Необхідно розробити консольний Java-застосунок для бронювання номерів у готелі.



У роботі потрібно:



\- створити класи предметної області;

\- виділити спільний ресурс;

\- реалізувати небезпечний варіант без синхронізації;

\- реалізувати безпечний варіант з використанням `synchronized`;

\- створити кілька потоків, які одночасно працюють зі спільним ресурсом;

\- використати `start()` для запуску потоків;

\- використати `join()` для очікування завершення потоків;

\- порівняти результати несинхронізованого і синхронізованого варіантів;

\- виміряти час виконання послідовного і паралельного сценаріїв;

\- написати модульні тести.



\## Опис предметної області



Предметна область роботи — бронювання номерів у готелі.



У готелі є певна кількість кімнат. Кожна кімната має обмежену місткість. Клієнти надсилають заявки на бронювання місць. У програмі заявки виконуються в різних потоках, тому кілька потоків можуть одночасно намагатися забронювати місця в одному й тому самому готелі.



\## Опис спільного ресурсу



Спільним ресурсом у програмі є об’єкт `HotelRoom`.



Основним змінюваним станом є поле:



```java

private int bookedPlaces;

```



Це поле зберігає кількість уже заброньованих місць у кімнаті. Кілька потоків можуть одночасно звертатися до цього поля через методи бронювання.



Також у програмі є сервіс `BookingService`, який містить список кімнат і обробляє заявки на бронювання.



\## Основні класи програми



\### `HotelRoom`



Клас описує кімнату готелю.



Він містить:



\- номер кімнати;

\- загальну місткість;

\- кількість заброньованих місць;

\- небезпечний метод бронювання `bookUnsafe()`;

\- безпечний метод бронювання `bookSafe()`.



\### `BookingRequest`



Клас описує заявку на бронювання.



Він містить:



\- номер заявки;

\- ім’я гостя;

\- кількість місць для бронювання.



\### `BookingService`



Клас відповідає за обробку заявок на бронювання.



Він містить список кімнат і методи:



\- `bookUnsafe()`;

\- `bookSafe()`;

\- `getTotalCapacity()`;

\- `getTotalBookedPlaces()`;

\- `getTotalFreePlaces()`.



\### `BookingTask`



Клас реалізує інтерфейс `Runnable`.



Об’єкт цього класу виконується в окремому потоці та багато разів надсилає заявки на бронювання.



\### `Main`



Демонстраційний клас програми.



У ньому створюються потоки, запускаються сценарії без синхронізації та з синхронізацією, а також виконується порівняння часу послідовного і паралельного виконання.



\## Небезпечний варіант без синхронізації



У класі `HotelRoom` реалізовано метод:



```java

public boolean bookUnsafe(int places)

```



Цей метод не використовує `synchronized`.



У ньому спочатку перевіряється, чи є вільні місця, а потім змінюється значення `bookedPlaces`.



Проблема полягає в тому, що операція бронювання не є атомарною. Вона складається з кількох дій:



1\. прочитати поточну кількість заброньованих місць;

2\. перевірити наявність вільних місць;

3\. обчислити нове значення;

4\. записати нове значення.



Якщо кілька потоків виконують ці дії одночасно, вони можуть прочитати однакове старе значення і записати неправильний результат.



\## Синхронізований варіант



Для коректної роботи реалізовано метод:



```java

public synchronized boolean bookSafe(int places)

```



У цьому методі використано `synchronized`, тому одночасно тільки один потік може виконувати бронювання для конкретного об’єкта `HotelRoom`.



Це захищає поле `bookedPlaces` від одночасної зміни кількома потоками.



\## Критична секція



Критичною секцією є фрагмент коду, у якому перевіряється кількість вільних місць і змінюється поле `bookedPlaces`.



```java

if (bookedPlaces + places > capacity) {

&#x20;   return false;

}



bookedPlaces += places;

```



Ця частина коду має виконуватися як єдина операція. Якщо її не синхронізувати, може виникнути некоректний результат.



\## Вибір монітора



У методі `bookSafe()` використано синхронізований метод.



Монітором є поточний об’єкт `HotelRoom`, тобто `this`.



Це означає, що для кожної окремої кімнати доступ до бронювання контролюється окремо. Якщо один потік бронює конкретну кімнату, інший потік повинен дочекатися завершення цієї операції.

\## Основні фрагменти коду



\### Небезпечне бронювання



```java

public boolean bookUnsafe(int places) {

&#x20;   checkPlaces(places);



&#x20;   if (bookedPlaces + places <= capacity) {

&#x20;       int currentBooked = bookedPlaces;

&#x20;       LockSupport.parkNanos(1L);

&#x20;       bookedPlaces = currentBooked + places;

&#x20;       return true;

&#x20;   }



&#x20;   return false;

}

```



Цей метод демонструє потенційно небезпечну роботу зі спільним ресурсом без синхронізації.



\### Безпечне бронювання



```java

public synchronized boolean bookSafe(int places) {

&#x20;   checkPlaces(places);



&#x20;   if (bookedPlaces + places > capacity) {

&#x20;       return false;

&#x20;   }



&#x20;   bookedPlaces += places;

&#x20;   return true;

}

```



Цей метод захищає критичну секцію за допомогою `synchronized`.



\### Запуск потоків



```java

for (int i = 0; i < THREAD\_COUNT; i++) {

&#x20;   tasks\[i] = new BookingTask(service, mode, iterations, "Гість-" + (i + 1));

&#x20;   threads\[i] = new Thread(tasks\[i], "Потік-" + (i + 1));

&#x20;   threads\[i].start();

}

```



У цьому фрагменті створюються і запускаються потоки.



\### Очікування завершення потоків



```java

for (Thread thread : threads) {

&#x20;   thread.join();

}

```



Метод `join()` використовується для того, щоб головний потік дочекався завершення всіх робочих потоків.



\## Параметри запуску експерименту



У програмі використано такі параметри:



```java

private static final int THREAD\_COUNT = 4;

private static final int ITERATIONS = 100\_000;

private static final int ROOM\_COUNT = 20;

private static final int PLACES\_PER\_ROOM = 5;

```



Це означає:



\- кількість потоків — `4`;

\- кількість ітерацій на один потік — `100000`;

\- кількість кімнат у демонстраційному готелі — `20`;

\- кількість місць в одній кімнаті — `5`.



Загальна місткість демонстраційного готелю:



```text

20 \* 5 = 100 місць

```



Загальна кількість спроб бронювання:



```text

4 \* 100000 = 400000 спроб

```



\## Результати виконання програми



Під час запуску програма виконує три основні частини:



1\. сценарій без `synchronized`;

2\. сценарій із `synchronized`;

3\. порівняння послідовного і паралельного часу виконання.



У небезпечному варіанті без синхронізації можливі некоректні результати через стан перегонів. Кілька потоків можуть одночасно прочитати старе значення `bookedPlaces` і записати неправильний результат.



У синхронізованому варіанті кількість фактично заброньованих місць не перевищує місткість готелю, тому що критична секція захищена методом `synchronized`.

\## Порівняння послідовного і паралельного виконання



У програмі реалізовано порівняння часу виконання для однакового обсягу роботи.



У послідовному сценарії всі заявки виконуються в одному потоці.



У паралельному сценарії заявки розподіляються між кількома потоками.



Паралельний варіант не завжди буде швидшим, тому що використання потоків і синхронізації створює додаткові накладні витрати. До них належать:



\- створення потоків;

\- перемикання між потоками;

\- очікування доступу до синхронізованого методу;

\- захоплення і звільнення монітора.



Тому результат часу потрібно пояснювати не тільки з точки зору швидкості, а й з точки зору коректності роботи програми.

\## Висновки



У ході виконання лабораторної роботи було розроблено консольний Java-застосунок для бронювання номерів у готелі.



Було показано проблему спільного доступу до змінюваного стану на прикладі поля `bookedPlaces`, яке змінюється кількома потоками. Небезпечний метод `bookUnsafe()` демонструє можливість некоректної роботи без синхронізації.



Для виправлення проблеми було реалізовано метод `bookSafe()` з використанням `synchronized`. Завдяки цьому критична секція виконується тільки одним потоком одночасно, а підсумковий стан ресурсу залишається коректним.



Також у роботі було використано `Thread`, `Runnable`, `start()` і `join()`, проведено порівняння часу послідовного та паралельного виконання, а також створено JUnit-тести для перевірки основної логіки програми.
